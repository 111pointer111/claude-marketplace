# 历史人物 Persona 蒸馏流水线

> 本文件为 OpenClaw 每日定时任务的执行规则。
> 目标：从原始史料自动蒸馏历史人物 persona，输出标准化 SKILL.md，上传至 GitHub 仓库。
> 状态管理：所有执行状态记录在 `DONE.md` 中，OpenClaw 自主读取并决定下一个处理人物。

---

## 一、流水线入口

### 触发机制

- **执行频率：** 每日北京时间 06:00
- **触发条件：** DONE.md 中「下一待处理」有有效条目，且该人物未在 done/ 中
- **并发限制：** 每次仅处理 1 个人物

### 入口操作（每次执行前必做）

```
1. 读取 DONE.md 中的「下一待处理」节
2. 确认人物名称
3. 读取 backlog.md 中该人物的完整条目（含 sources 列表）
4. 检查 done/{姓名}.done 是否存在 → 存在则跳过此人，更新「下一待处理」为下一个 priority high 人物
5. 若 backlog.md 中所有 priority high 人物均已完成，重新计算下一个最高优先级人物
6. 若 backlog.md 中所有人均已完成 → 流水线终止，记录"DONE.md 全量完成"
```

### 人物选择规则

```
优先级顺序（由高到低）：
1. priority: high + status: pending + backlog 中排在最前
2. priority: medium + status: pending + backlog 中排在最前
3. priority: low + status: pending + backlog 中排在最前

status: done 的人 → 永不重复处理
status: failed 的人 → 保留在 backlog 中，标记 status: failed，等待人工介入
```

## 二、Stage 1：资料采集

**目标：** 获取人物的原始文本语料。

**执行步骤：**

```
1.1 读取 backlog.md 中该人物的 sources 列表
1.2 对每个 source URL 执行以下操作：

    IF type == "poetry_collection" OR type == "collected_works" OR type == "personal_essay":
        → 使用 WebFetch 抓取全文原始文本
        → 保存到 raw/{人物名}/{类型}_{序号}.txt
        → 标注每段/每首诗的写作年份（如可考证）

    IF type == "historical_record":
        → 抓取传记正文
        → 保存到 raw/{人物名}/传记.txt
        → 提取其中直接引语（用引号标记），单独保存到 raw/{人物名}/引语.txt

    IF type == "commentary":
        → 抓取后人评述
        → 保存到 raw/{人物名}/后世评述.txt

1.3 补充搜索（如果原始 sources 不够丰富，诗词 < 30 首）：
    → 执行 WebSearch: "{人物名} 诗词全集", "{人物名} 文集", "{人物名} 书信"
    → 从搜索结果中补充 1-2 个权威来源（中华书局/古诗文网/中国哲学书电子化计划）
    → 保存到 raw/{人物名}/补充_{序号}.txt
```

**输出目录结构：**
```
raw/
└── 苏轼/
    ├── 诗词_1.txt           # 含年份标注
    ├── 文集.txt
    ├── 东坡志林.txt
    ├── 传记.txt
    ├── 引语.txt              # 直接引语
    └── 后世评述.txt
```

**质量标准：**
- 诗词语料 ≥ 50 首（不足 30 首则 confidence 上限为 medium）
- 文集/散文语料 ≥ 20 篇
- 有传记依据
- 全部语料合并为 raw/{人物名}/全部语料.txt 供后续使用

## 三、Stage 2：阶段切分

**目标：** 将人物一生划分为若干阶段。

**执行步骤：**

```
2.1 读取 raw/{人物名}/传记.txt 和诗词年份标注
2.2 将一生划分为 2-5 个阶段，划分依据：
    ├── 重大人生转折点（贬谪/升迁/战乱/亡国等）
    ├── 创作风格明显变化的节点
    └── 年龄节点（通常：早年 0-30，中年 30-55，晚年 55+）

2.3 对每个阶段输出：
    - 起止年份
    - 主要生活地点
    - 该阶段核心事件（3-5条）
    - 代表作品列表
    - 该阶段语言风格概述
    - confidence: high（现存作品 ≥ 30篇）/ medium（10-29篇）/ low（< 10篇）

2.4 输出文件：processed/{人物名}/stages.md
```

## 四、Stage 3：多维蒸馏

---

### 4.1 思想内核（跨时期不变的核心）

```
分析指令：
你是一位古典文学与思想史专家。请从以下语料中提炼出"{人物名}"的思想内核。
思想内核是这个人在任何时期、任何作品中都保持不变的核心价值观与底线。

请输出 JSON：
{
  "核心价值观": [{"条目": "...", "原文引用": [...], "出处": "..."}],
  "政治立场": {"描述": "...", "原文引用": [...]},
  "底线清单": [{"条目": "...", "原文引用": "...", "出处": "..."}],
  "自我定位": {"描述": "...", "原文引用": [...]}
}

要求：每条结论至少附 2 处原文引用，必须可从 raw/{人物名}/ 原始文本回溯。
```

**输出：** `processed/{人物名}/dimension_思想内核.json`

---

### 4.2 语言特征（量化评分，最成熟的维度）

```
分析指令：
你是一位古典文学风格分析专家。请分析以下{人物名}的作品，提取语言特征。

分析范围：诗词 {N} 首 + 文 {M} 篇

请提取以下维度，每维度给出评分（1-5）和原文佐证：

1. 文言/白话光谱（1=全文言，5=大量白话入诗）
2. 雅/俗光谱（1=极典雅，5=俚语民谣风格）
3. 骈偶 vs 散行（骈%估算）
4. 句长偏好：短句(≤7字)% / 中句(8-15字)% / 长句(≥16字)%
5. 语气标签（可多选）：慷慨/沉郁/清淡/诙谐/悲凉/豪迈/婉约/冷峻/炽热/隐忍
6. 高频意象 Top 10
7. 修辞偏好（可多选）：用典/白描/比喻/对偶/夸张/反问/借代/比兴
8. 文体偏好排序：诗/词/文/赋/书信/奏议

输出格式：JSON（便于程序处理）
```

**输出：** `processed/{人物名}/dimension_语言特征.json`

---

### 4.3 表达偏好（对外表达策略）

```
分析指令：
你是一位修辞学专家。请分析"{人物名}"对不同对象的表达策略差异。

请分析：
1. 对**上位者**（君王/权贵）→ 直谏/婉谏/讽刺/沉默？附原文
2. 对**友人**（平辈/知己）→ 戏谑/真挚/克制？附原文
3. 对**敌人/政敌** → 犀利/绵里藏针/回避？附原文
4. 对**百姓/民生** → 平实/悲悯/激愤？附原文
5. 对**自己**（内心独白/自嘲）→ 风格？附原文

无直接语料时，标注"[无直接语料，据诗文推测]"。
输出格式：JSON
```

**输出：** `processed/{人物名}/dimension_表达偏好.json`

---

### 4.4 立场光谱

```
分析指令：
请为"{人物名}"在以下议题上打分。
评分：-2（极端反对）到 +2（极端支持），0=中立/无关

议题：皇权专制 / 变法革新 / 民生疾苦 / 华夷之辨 / 文学功利观 / 出世倾向(道家) / 入世倾向(儒家)

每条附原文依据。输出格式：JSON立场向量
```

**输入：** raw/{人物名}/全部语料.txt + raw/{人物名}/后世评述.txt
**输出：** `processed/{人物名}/dimension_立场光谱.json`

---

### 4.5 阶段一致性校验

```
分析指令：
请对比"{人物名}"不同人生阶段的语言风格与思想倾向：

1. 哪些特征在所有阶段保持一致？（核心风格不变的部分）
2. 哪些特征随阶段明显变化？（发展的部分）
3. 思想内核在所有阶段是否保持一致？
   如有矛盾，判断：a)"史料误差" 或 b)"人物真实思想发展"，附判断依据
4. 每个阶段的 confidence 评级是否合理？

输出：一致性报告（markdown）+ 最终 confidence 调整建议
```

**输出：** `processed/{人物名}/consistency_check.md`

## 五、Stage 4： Persona Card 生成

**目标：** 将所有维度整合为标准化文件。

**执行步骤：**

```
5.1 读取 processed/{人物名}/ 所有维度文件
5.2 生成 output/{人物名}/SKILL.md（按 Stage 5 模板）
5.3 生成 output/{人物名}/README.md（人物简介 + 使用说明）
5.4 生成 output/{人物名}/METADATA.json（元数据）
5.5 生成 output/{人物名}/CITATIONS.md（所有原文引用，清单格式）
5.6 生成 output/{人物名}/raw_stats.json（统计：处理诗词N首/文M篇/引用X条）
5.7 创建空文件 done/{人物名}.done（标记完成）
```

## 七、质量控制

### 自动校验规则

```
规则1: 引用可回溯
→ SKILL.md 中每一处引用，必须能在 raw/{人物名}/ 原始文本中找到对应原文
→ 任何引用无法回溯 → 标记 WARN，输出时注明"[引用来源待核实]"

规则2: 语言特征与原文一致
→ 维度评分(1-5)与佐证文本是否匹配？
→ 不一致 → 标记 WARN，人工复核

规则3: 置信度守门
→ 诗词 < 30 首 → overall_confidence ≤ medium
→ 诗词 < 10 首 → overall_confidence = low
→ 某阶段作品 < 10 篇 → 该阶段 confidence = low

规则4: 成语/典故验证
→ SKILL.md 中出现的成语典故，执行一次 WebSearch 验证来源
→ 错引 → 标记 WARN 并修正

规则5: 自我矛盾处置
→ 若人物在不同时期存在立场矛盾：
    a) 优先判断为"史料误差" → 标注并排除
    b) 确为"人物真实发展" → 如实呈现，标注"反映思想发展"
    c) 不强制统一，保留复杂性
```

### 输出前强制检查清单

```
□ 所有引用均可从 raw/ 目录回溯
□ 每条维度结论至少有 1 处原文引用支撑
□ overall_confidence 符合统计规则
□ 立场光谱中每个议题至少有 1 处原文依据
□ SKILL.md 正文（不含引用）字数在 1500-5000 字之间
□ done/{人物名}.done 已创建
□ GitHub 已 push 成功
□ DONE.md 已更新（追加完成记录 + 更新下一待处理 + 更新统计）
□ backlog.md 中该人物 status 已改为 done
```

## 八、GitHub 上传与 DONE.md 更新

### 执行顺序

```
8.1 生成 output/{人物名}/ 所有文件
8.2 Git 操作
    → git add output/{人物名}/
    → git add done/{人物名}.done
    → git commit -m "distill: {人物名} persona ({confidence})"
    → git push origin main

8.3 Git push 成功后，更新 DONE.md：
    a) 在「已完成」表格追加一行
    b) 更新「统计概览」
    c) 在 backlog.md 中将人物 status 改为 done
    d) 从 backlog.md 读取下一个最高优先级 pending 人物
    e) 更新「下一待处理」节

8.4 错误处理
    → 若 git push 失败，保留本地缓存，次日重试
    → 重试 3 次仍失败 → 在 DONE.md「执行日志」中记录失败，
      backlog.md 中该人物 status 改为 failed，等待人工介入
    → 若 Git push 成功但 DONE.md 更新失败 → 单独提交 DONE.md 更新
```

## 九、人物队列（backlog.md 格式）

```markdown
| 人物 | 朝代 | priority | status | 备注 |
|------|------|----------|--------|------|
| 苏轼 | 北宋 | high | pending | |
| 杜甫 | 唐 | high | pending | |
| ... | ... | ... | ... | |
```

**Priority 定义：**
- high：史料极丰富（诗词100+首，文集完整，传记详尽），可生成完整版
- medium：史料较丰富（诗词30-100首），可生成标准版
- low：史料有限（诗词<30首），生成精简版

**Status 定义：**
- pending：待处理
- done：已完成
- failed：执行失败，需人工介入
